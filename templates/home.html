Requirement Writing Guide

✓ DO's - Write Clear Requirements

  • Use action verbs: "Users should be able to register", "System must validate email"
  • Be specific: "Password must contain 8+ characters, 1 uppercase, 1 number, 1 special character"
  • Include constraints: "Maximum 5 items per cart", "Minimum 10 characters"
  • Mention error cases: "If payment fails, display error message"
  • Use consistent format: Bullet points or numbered lists work best
  • Include validation rules: "Email must be unique and valid format"
  • Specify conditions: "Users can only login after email verification"
  • Mention business rules: "Generate confirmation number after successful order"

✗ DON'Ts - Avoid These Mistakes

  • Avoid vague statements: "System should work properly"
  • Don't use ambiguous language: "User should be able to do something"
  • Avoid implicit assumptions: Assume nothing about domain knowledge
  • Don't mix multiple requirements: Keep one requirement per statement
  • Avoid generic descriptions: "The system should be good"
  • Don't skip validation rules: Always mention format/length requirements
  • Avoid unclear abbreviations: Use full terms

📋 Template Examples

✓ Functional Requirement:

"Users should be able to register for an account using their email address and password. The email address must be unique and follow valid email format. The password must contain at least 8 characters, including one uppercase letter, one number, and one special character."

✓ Business Rule:

"After successful registration, users should receive a verification email containing a link to activate their account. Users should only be able to login after verifying their email address."

✓ Error Scenario:

"If a user enters an incorrect password three times consecutively, the account should be temporarily locked for 10 minutes. After 10 minutes, the user should be able to attempt login again."

🎯 Requirement Quality Checklist

  • ☐ Requirement has clear action verb (create, register, validate, etc.)
  • ☐ Requirement specifies what entity is affected (user, account, email, etc.)
  • ☐ Requirement includes validation rules or constraints
  • ☐ Requirement mentions error scenarios or edge cases
  • ☐ Requirement is not vague or ambiguous
  • ☐ Requirement is specific and measurable
  • ☐ Requirement is independent (not dependent on other requirements)
  • ☐ Requirement uses consistent terminology
